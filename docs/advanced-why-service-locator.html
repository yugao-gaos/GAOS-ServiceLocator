Why Service Locator?

Understanding why GAOS Service Locator is the best choice for Unity projects

Pattern Comparison

Singleton Pattern

The traditional approach to global state management in Unity.

Advantages

  • Simple to implement
  • Direct access to services
  • No configuration needed

Disadvantages

  • Hard to test (global state)
  • Tight coupling
  • No lifecycle management
  • No dependency validation
  • Hard to maintain

Dependency Injection

The enterprise-grade solution for dependency management.

Advantages

  • Loose coupling
  • Easy to test
  • Clear dependencies
  • Compile-time validation

Disadvantages

  • Complex setup
  • Steep learning curve
  • Boilerplate code
  • Runtime overhead
  • Not Unity-friendly

Traditional Service Locator

The middle ground between Singleton and DI.

Advantages

  • Flexible service resolution
  • Runtime configuration
  • Easier testing than Singletons
  • Simple to understand

Disadvantages

  • Hidden dependencies
  • Runtime resolution errors
  • No validation
  • Potential misuse

GAOS Service Locator

A modern, Unity-optimized service management solution.

Advantages

  • Unity-first design with full lifecycle support
  • Automatic service discovery and registration
  • Async service initialization and resolution
  • Editor-time validation and debugging tools
  • Zero runtime reflection overhead

Considerations

  • Services need attribute decoration
  • Initial setup required for custom services

Our Approach

Combines the best of both worlds while solving common issues:

From DI Pattern

  • Compile-time validation
  • Clear dependency graphs
  • Proper lifetime management

From Service Locator

  • Simple, intuitive API
  • Runtime flexibility
  • Lightweight implementation

Added Benefits

  • Unity-optimized architecture
  • Async service support
  • Rich debugging tools
