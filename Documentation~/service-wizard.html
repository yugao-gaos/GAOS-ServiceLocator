1. Does your service need to maintain state?

State refers to data that persists between service calls (e.g., settings, cached data, current status)

Example: A GameSettingsService that stores volume levels needs state, while a MathService that just performs calculations doesn't.

2. How does your service interact with Unity?

Choose the level of Unity integration your service requires

Example: An InputManager needs scene interaction for Update() calls, while a DataValidator works without Unity integration.

3. When should the service be available?

Choose when your service needs to operate

Example: A LevelEditorService is EditorOnly, while a PlayerProgressService needs to run during gameplay (Runtime).

4. How should instances be managed?

Choose how service instances should be created and managed

Example: GameSettingsService can be Singleton (shared across app), LevelManagerService as SceneSingleton (one per scene), LocalizationService as Transient (new per request), or EnemyFactoryService as SceneTransient (new per request within scene).
