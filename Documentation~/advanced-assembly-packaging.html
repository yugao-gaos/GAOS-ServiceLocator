Understanding Assembly Definitions

Why Use Assembly Definitions?

Assembly Definitions provide several key benefits for service organization and management:

  • Improved compilation times through parallel compilation
  • Better encapsulation of service implementations
  • Clear dependency boundaries between service modules
  • Reduced memory usage through selective assembly loading

Key Benefits

  • Modularity: Isolate services and their dependencies
  • Performance: Faster compilation and runtime loading
  • Maintainability: Clear boundaries and dependencies
  • Reusability: Package services for reuse across projects

Custom Packages

Benefits of Package Organization

Organizing services into custom packages offers additional advantages:

  • Version control and dependency management
  • Easy sharing between projects
  • Standardized distribution format
  • Integration with Unity's Package Manager

Cross-Assembly Service Usage

Managing Service Dependencies

When using services across different assemblies, consider these patterns:

// Assembly: Company.Core.Interfaces
public interface IDataService { }

// Assembly: Company.Core.Services
[Service(typeof(IDataService), "Data")]
public class DataService : IDataService { }

// Assembly: Company.Game
public class GameManager
{
    private readonly IDataService _dataService;
    
    public GameManager()
    {
        _dataService = ServiceLocator.GetService();
    }
}

Best Practices

  • Keep interface assemblies lightweight
  • Avoid circular assembly dependencies
  • Use assembly definition references for Unity-specific dependencies
  • Consider runtime performance when designing assembly boundaries

Interface Organization Approaches

Central Interface Layer

All service interfaces in a single assembly

Advantages
  • Single reference for all consumers
  • Consistent versioning
  • Easy to maintain API compatibility
  • Clear service contract overview
Disadvantages
  • Potential unnecessary dependencies
  • Less granular versioning
  • May grow too large over time
// Assembly: Company.Services.Interfaces
namespace Company.Services
{
    public interface IDataService { }
    public interface INetworkService { }
    public interface IAnalyticsService { }
}

Per-Package Interfaces

Each package contains its own interface assembly

Advantages
  • Better encapsulation
  • Independent versioning
  • Minimal dependency overhead
  • More flexible deployment
Disadvantages
  • More assemblies to manage
  • Potential interface duplication
  • More complex dependency graph
// Assembly: Company.Data.Interfaces
namespace Company.Data
{
    public interface IDataService { }
}

// Assembly: Company.Network.Interfaces
namespace Company.Network
{
    public interface INetworkService { }
}

Choosing an Approach

Consider these factors when deciding:

  • Project Size: Smaller projects benefit from central interfaces, larger ones from per-package
  • Team Structure: Multiple teams might prefer per-package for independence
  • Deployment: Need for separate deployment of different service modules
  • Evolution: Expected rate of change and versioning needs
